## 📦 Feature Breakdown

### 🔥 Priority 1: Lambda/Closure Support (Weeks 1-3)

**Current State**: Lambda generates null pointer stub, causing undefined reference errors.

**Implementation Plan**:

#### Phase 1.1: Environment Analysis (Week 1)
- [ ] **Capture Variable Detection**
  - Walk lambda body AST, identify non-local variables
  - Build capture list: `{x: Integer, y: String}`
  
- [ ] **Environment Struct Generation**
  ```c
  // For: var f := (a: Integer) -> x + a;  // captures x
  %__env_f = type { i64 }  // struct { x: i64 }
  ```

- [ ] **Test**: Simple capture (1 variable, primitive type)

#### Phase 1.2: Lambda Lowering (Week 2)
- [ ] **Function Pointer Generation**
  ```llvm
  ; Lambda body becomes named function
  define i64 @__lambda_f(ptr %env, i64 %a) {
    %x_ptr = getelementptr %__env_f, ptr %env, i32 0, i32 0
    %x = load i64, ptr %x_ptr
    %sum = add i64 %x, %a
    ret i64 %sum
  }
  ```

- [ ] **Environment Allocation**
  ```llvm
  ; At lambda creation site
  %env = call ptr @malloc(i64 8)  ; sizeof(__env_f)
  %x_slot = getelementptr %__env_f, ptr %env, i32 0, i32 0
  store i64 %x, ptr %x_slot
  ```

- [ ] **Closure Struct**
  ```llvm
  %closure = alloca { ptr, ptr }  ; {func_ptr, env_ptr}
  %fptr = getelementptr {ptr, ptr}, ptr %closure, i32 0, i32 0
  store ptr @__lambda_f, ptr %fptr
  %eptr = getelementptr {ptr, ptr}, ptr %closure, i32 0, i32 1
  store ptr %env, ptr %eptr
  ```

- [ ] **Test**: Call lambda with captured variable

#### Phase 1.3: Advanced Cases (Week 3)
- [ ] **Multiple captures** (`var f := (a) -> x + y + a`)
- [ ] **Nested lambdas** (closure inside closure)
- [ ] **Mutable captures** (write to captured variable)
- [ ] **Lambda as return value** (escape analysis)

**Deliverables**:
- 15+ tests in `pkg/llvmgen/lambda_test.go`
- example15_lambda.klx passes
- Documentation: `docs/llvm-backend.md` updated (remove lambda limitation)

### 🔧 Priority 2: Complete Multi-Return Tuple Destructuring (Week 4)

**Current State**: `(a, b) := func()` generates comment stub, variables uninitialized.

**Implementation Plan**:

#### Step 2.1: Function Return Struct
- [ ] Detect multi-return in function signature
  ```pascal
  function DivMod(n, d: Integer): (Integer, Integer);
  ```
  
- [ ] Generate return struct type
  ```llvm
  %__ret_DivMod = type { i64, i64 }
  
  define %__ret_DivMod @DivMod(i64 %n, i64 %d) {
    %q = sdiv i64 %n, %d
    %r = srem i64 %n, %d
    %ret = insertvalue %__ret_DivMod undef, i64 %q, 0
    %ret2 = insertvalue %__ret_DivMod %ret, i64 %r, 1
    ret %__ret_DivMod %ret2
  }
  ```

#### Step 2.2: Tuple LHS Destructuring
- [ ] Parse `(a, b) := expr` correctly (already done in AST)
  
- [ ] Codegen for destructuring assignment
  ```llvm
  %call_ret = call %__ret_DivMod @DivMod(i64 17, i64 5)
  %q_val = extractvalue %__ret_DivMod %call_ret, 0
  %r_val = extractvalue %__ret_DivMod %call_ret, 1
  store i64 %q_val, ptr %v_q_int
  store i64 %r_val, ptr %v_r_int
  ```

- [ ] Handle nested tuples (optional, low priority)

**Deliverables**:
- 8+ tests in `pkg/llvmgen/multireturn_test.go`
- example16_multireturn.klx works correctly (not just compiles)
- Documentation update

### 🏗️ Priority 3: `inherited` Keyword Support (Week 5)

**Current State**: Not implemented. Workaround: explicit parent method calls.

**Implementation Plan**:

#### Step 3.1: Parent Method Lookup
- [ ] In method body, detect `inherited;` or `inherited.MethodName(args)`
  
- [ ] Walk class hierarchy to find parent implementation
  ```go

    // Traverse g.classes[className].Parent chain
  }
  ```

#### Step 3.2: Codegen for `inherited`
- [ ] Bare `inherited;` (same method name, same args)
  ```llvm
  ; In TChild.DoSomething override:
  call void @TParent_DoSomething(ptr %this, ...)
  ```

- [ ] `inherited.SpecificMethod(args)`
  ```llvm

  ```

- [ ] Correct `this` pointer passing (no vtable indirection)

### ⚡ Priority 4: Optimization Passes (Week 6)

**Current State**: No optimization, code is 2-5x slower than Go backend.

**Implementation Plan**:

#### Step 4.1: Enable LLVM opt Tool
- [ ] Add `--llvm-opt` flag (already sketched in M1/M2)
  ```bash
  kylix build --backend=llvm --llvm-opt=2 program.klx
  # Runs: opt -O2 program.ll -o program_opt.ll
  ```

- [ ] Default optimization level: `-O1` (basic inlining + DCE)
- [ ] Advanced: `-O2` (aggressive inlining, loop opts)
- [ ] Experimental: `-O3` (vectorization, may break some code)

#### Step 4.2: Custom Optimization Hints
- [ ] Inline pragma (optional)
  ```pascal
  {$inline on}
  function FastAdd(a, b: Integer): Integer;
  ```
  → generates `alwaysinline` attribute

- [ ] Hot/cold path hints (low priority)

#### Step 4.3: Benchmarking
- [ ] Create benchmark suite: `benchmarks/llvm/`
  - Fibonacci (recursion test)
  - Prime sieve (loop test)
  - String manipulation (allocation test)
  
- [ ] Measure: LLVM unopt vs LLVM opt vs Go backend
- [ ] Target: LLVM O2 within 1.5x of Go backend

**Deliverables**:
- `--llvm-opt` flag functional
- Benchmark results documented in `docs/llvm-performance.md`
- At least 30% speedup vs unoptimized

## 🐛 Known Issues & Risks

### Technical Risks

1. **Closure memory leaks**
   - **Risk**: Environment structs malloc'd but never freed (no GC)
   - **Mitigation**: Document as limitation, defer GC to v5.0
   - **Workaround**: Use named functions for long-running programs

2. **Optimization breaks exception handling**
   - **Risk**: LLVM passes might eliminate setjmp/longjmp calls
   - **Mitigation**: Mark exception functions with `noinline` attribute
   - **Test**: Run exception tests with `-O2` enabled

3. **Multi-return ABI conflicts**
   - **Risk**: Struct return calling convention varies by platform
   - **Mitigation**: Use explicit return struct type (already in plan)

### Schedule Risks

- **Lambda complexity underestimated**: Add 1 week buffer (Week 7)
- **Optimization debugging takes longer**: Scale back to `-O1` only

## 🔮 Looking Ahead: v4.2.0 Ideas

After v4.1.0, consider:

1. **LLVM stdlib Phase 1** — Compile core stdlib modules with LLVM backend
   - `strutil`, `mathutil`, `sysutil` (pure Kylix, no Go wrapper)
   - Goal: Reduce dependency on Go stdlib wrappers

2. **Incremental compilation** — Cache LLVM IR per module
   - Only recompile changed files
   - Link pre-compiled .o files

3. **Debug symbols** — Emit DWARF debug info
   - Enable GDB/LLDB stepping through Kylix source
   - `kylix build --backend=llvm -g program.klx`

4. **Cross-compilation** — Build for different targets without LLVM installed
   - Pre-compile LLVM IR, ship with linker only
   - `kylix build --backend=llvm --target=aarch64-linux-gnu`
